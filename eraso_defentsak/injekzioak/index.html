    2.1 Defensas contra inyecciones¶

    Inyección SQL¶

    ¿Qué es la inyección SQL?¶

    La inyección SQL es un tipo de ataque que modifica una consulta a la base de datos, lo que ocurre cuando la entrada del usuario se inserta en la consulta sin la validación adecuada.

    ¿Cómo funciona?¶

    -- Entrada del usuario: ' OR '1'='1
-- Consulta original:
SELECT * FROM users WHERE username = '' AND password = 'contraseña'

-- Después de la inyección:
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = 'contraseña'
-- Como '1'='1' siempre es verdadero, la consulta devolverá todos los usuarios

    ¿Cómo prevenirlo?¶

    1. Usar consultas parametrizadas (Prepared Statements)¶

    Ejemplo en Python (SQLite):

    import sqlite3

def login_seguro(usuario, contraseña):
    conn = sqlite3.connect('base_de_datos.db')
    cursor = conn.cursor()

    # Usando consulta parametrizada
    consulta = "SELECT * FROM users WHERE username = ? AND password = ?"
    cursor.execute(consulta, (usuario, contraseña))

    usuario = cursor.fetchone()
    conn.close()
    return usuario

    Ejemplo en PHP (PDO):

    <?php
$usuario = $_POST['usuario'];
$contraseña = $_POST['contraseña'];

$pdo = new PDO('mysql:host=localhost;dbname=mi_bd', 'usuario', 'contraseña');
$stmt = $pdo->prepare('SELECT * FROM users WHERE username = :usuario AND password = :contraseña');
$stmt->execute(['usuario' => $usuario, 'contraseña' => $contraseña]);
$usuario = $stmt->fetch();
?>

    XSS (Cross-Site Scripting)¶

    Tipos de XSS¶

    1. XSS Reflejado (Reflected)
    2. El atacante envía un enlace malicioso a la víctima
    3. El sitio web refleja el script en la respuesta

    4. El script se ejecuta en el navegador de la víctima

    5. XSS Almacenado (Stored)

    6. El script malicioso se guarda en la base de datos
    7. Se ejecuta cuando un usuario carga la página

    8. Puede causar más daño ya que afecta a múltiples usuarios

    9. XSS basado en DOM

    10. Ocurre únicamente en el navegador
    11. Modifica el Modelo de Objetos del Documento (DOM)

    ¿Cómo prevenir XSS?¶

    1. Codificar la salida¶

    JavaScript (Node.js):

    const escapeHtml = (texto) => {
  const mapa = {
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    '"': '&quot;',
    "'": '&#x27;',
    "/": '&#x2F;'
  };
  return texto.replace(/[&<>"'/]/g, caracter => mapa[caracter]);
};

// Uso
const entradaPeligrosa = "<script>código</script>";
document.getElementById('nombre').textContent = escapeHtml(entradaPeligrosa);

    2. Usar APIs seguras¶

    JavaScript seguro:

    // Incorrecto
document.getElementById('nombre').innerHTML = entradaUsuario;

// Correcto
document.getElementById('nombre').textContent = entradaUsuario;

    3. Usar Content Security Policy (CSP)¶

    Content-Security-Policy: default-src 'self'; script-src 'self' https://apis.google.com

    Otras inyecciones¶

    Inyección de comandos¶

    Ejemplo peligroso (Python):

    import os

def hacer_ping(direccion):
    # Peligroso: la entrada del usuario puede ejecutar comandos
    os.system(f"ping -c 4 {direccion}")

    Seguro:

    import subprocess

def hacer_ping_seguro(direccion):
    # Seguro: los parámetros se envían como lista
    subprocess.run(['ping', '-c', '4', direccion], check=True)

    Inyección LDAP¶

    Ejemplo peligroso:

    import ldap

def autenticar(usuario, contraseña):
    # Peligroso: entrada sin validar
    consulta = f"(uid={usuario})(userPassword={contraseña})"
    # Realizar consulta LDAP

    Seguro:

    import ldap
from ldap.filter import escape_filter_chars

def autenticar_seguro(usuario, contraseña):
    # Caracteres especiales escapados
    usuario_limpio = escape_filter_chars(usuario)
    pass_limpio = escape_filter_chars(contraseña)
    consulta = f"(uid={usuario_limpio})(userPassword={pass_limpio})"
    # Realizar consulta LDAP
