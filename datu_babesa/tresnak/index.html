    3.3 Herramientas prácticas de seguridad¶

    Desarrollo Seguro (DevSecOps)¶

    1. SAST (Análisis de Seguridad de Aplicaciones Estático)¶

    Con el plugin de seguridad para ESLint (JavaScript/TypeScript)

    # Instalar paquetes necesarios
npm install --save-dev eslint eslint-plugin-security

    Configuración de .eslintrc.js:

    module.exports = {
  "extends": [
    "eslint:recommended",
    "plugin:security/recommended"
  ],
  "plugins": ["security"],
  "rules": {
    "security/detect-buffer-noassert": "error",
    "security/detect-child-process": "error",
    "security/detect-disable-mustache-escapes": "error",
    "security/detect-eval-with-expression": "error",
    "security/detect-no-csrf-before-method-override": "error",
    "security/detect-non-literal-fs-filename": "warn",
    "security/detect-non-literal-regexp": "error",
    "security/detect-non-literal-require": "warn",
    "security/detect-object-injection": "off", // Activar solo si es necesario, requiere cambios en el código
    "security/detect-possible-timing-attacks": "error",
    "security/detect-pseudoRandomBytes": "error",
    "security/detect-unsafe-regex": "error"
  }
};

    2. DAST (Análisis de Seguridad de Aplicaciones Dinámico)¶

    Usando OWASP ZAP (a través de Docker):

    # Iniciar OWASP ZAP en Docker
# Interfaz web: http://localhost:8080
# API: http://localhost:8081

docker run -u zap -p 8080:8080 -p 8081:8081 \
  -i owasp/zap2docker-stable zap-webswing.sh

# Escaneo automatizado
# (A través de la línea de comandos de ZAP)
docker run -i owasp/zap2docker-stable zap-baseline.py \
  -t https://tu-aplicacion.com \
  -r informe-pruebas.html \
  -w informe-zap.md \
  -l INFO

    3. SCA (Software Composition Analysis)¶

    Dependabot configuración en GitHub (.github/dependabot.yml):

    version: 2
updates:
  # JavaScript/Node.js actualizaciones
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    # Actualizaciones de seguridad inmediatas
    pull-request-branch-name:
      separator: "-"
    labels:
      - "dependencies"
      - "javascript"
    # Análisis de seguridad con OpenSSF Scorecard
    open-pull-requests-limit: 10
    # Actualizaciones basadas en boletines de seguridad
    vulnerability-alerts:
      enabled: true

  # Actualizaciones de imágenes de Docker
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "weekly"
    labels:
      - "dependencies"
      - "docker"

    Pruebas de seguridad automatizadas¶

    1. Nikto (Web Vulnerability Scanner)¶

    # Instalación (Ubuntu/Debian)
sudo apt install nikto

# Uso básico
nikto -h https://tu-aplicacion.com

# Guardar informe en formato HTML
nikto -h https://tu-aplicacion.com -o nikto_scan.html -F htm

# Uso de proxy (por ejemplo, OWASP ZAP)
nikto -h https://tu-aplicacion.com -useproxy http://localhost:8080

# Agregar cookie para seguir la sesión
nikto -h https://tu-aplicacion.com -C "sessionid=abc123; secure;"
nikto -h https://zure-aplikazioa.eus -C "sessionid=abc123; secure;"

    2. Nmap (Escáner de red)¶

    # Encontrar puertos abiertos
nmap -sS -p- -T4 -v tu-servidor.com

# Detectar servicios y versiones
nmap -sV -sC -p 80,443,8080 tu-servidor.com

# Usando NSE (Nmap Scripting Engine)
nmap --script=http-security-headers,http-csrf,http-sql-injection -p 80,443 tu-servidor.com

# Guardar salida en formato XML
nmap -oX resultados_escaner.xml -sV -sC tu-servidor.com

    Análisis Estático de Código¶

    1. SonarQube + SonarLint¶

    Configuración de SonarQube (Docker):

    # docker-compose.yml
version: '3'
services:
  sonarqube:
    image: sonarqube:community
    ports:
      - "9000:9000"
    environment:
      - SONAR_ES_BOOTSTRAP_CHECKS_DISABLE=true
      - SONAR_JDBC_URL=jdbc:postgresql://db:5432/sonar
      - SONAR_JDBC_USERNAME=sonar
      - SONAR_JDBC_PASSWORD=sonar
    volumes:
      - sonarqube_data:/opt/sonarqube/data
      - sonarqube_extensions:/opt/sonarqube/extensions
    depends_on:
      - db

  db:
    image: postgres:13
    environment:
      - POSTGRES_USER=sonar
      - POSTGRES_PASSWORD=sonar
      - POSTGRES_DB=sonar
    volumes:
      - postgresql_data:/var/lib/postgresql/data
      - postgresql_data_db:/var/lib/postgresql/data/db

volumes:
  sonarqube_data:
  sonarqube_extensions:
  postgresql_data:
  postgresql_data_db:

    Uso de Sonar-scanner (en cada proyecto):

    # Archivo sonar-project.properties
sonar.projectKey=nombre-del-proyecto
sonar.projectName="Nombre del Proyecto"
sonar.projectVersion=1.0
sonar.sources=.
sonar.sourceEncoding=UTF-8
sonar.exclusions=**/node_modules/**,**/*.spec.js
sonar.javascript.lcov.reportPaths=coverage/lcov.info

# Ejecutar análisis
docker run --rm \
  -v "$(pwd):/usr/src" \
  -w /usr/src \
  sonarsource/sonar-scanner-cli

    2. Snyk (Vulnerabilidades en Dependencias)¶

    # Instalar CLI de Snyk
npm install -g snyk

# Autenticarse con Snyk
snyk auth

# Analizar un proyecto
snyk test

# Monitorear dependencias
snyk monitor

# Analizar una imagen de Docker
snyk container test ubuntu:latest

# Análisis de Infraestructura como Código (Terraform, CloudFormation)
snyk iac test

    Automatización de Pruebas de Seguridad (CI/CD)¶

    Ejemplo de GitHub Actions¶

    # .github/workflows/security-scan.yml
name: Análisis de Seguridad

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]
  schedule:
    - cron: '0 0 * * 0' # Semanalmente los domingos

jobs:
  analisis-seguridad:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3

    # Configurar entorno Node.js
    - name: Usar Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18.x'
        cache: 'npm'

    # Instalar dependencias
    - name: Instalar dependencias
      run: npm ci

    # Ejecutar pruebas
    - name: Ejecutar pruebas
      run: npm test

    # Ejecutar linter
    - name: Ejecutar linter
      run: npx eslint . --ext .js,.jsx,.ts,.tsx

    # Análisis de seguridad con Snyk
    - name: Analizar vulnerabilidades con Snyk
      uses: snyk/actions/node@master
      env:
        SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
      with:
        args: --severity-threshold=high

    # Escaneo con OWASP ZAP (usando Docker)
    - name: Escaneo OWASP ZAP
      uses: zaproxy/action-baseline@v0.6.0
      with:
        target: 'https://tu-aplicacion.com'
        rules_file_name: '.zap/rules.tsv'
        risk: '1,2,3'  # 1: Informativo, 2: Peligroso, 3: Crítico
        fail_action: true  # Fallar el workflow si hay errores

    # Análisis de vulnerabilidades en imágenes Docker con Trivy
    - name: Analizar vulnerabilidades con Trivy
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: 'tu-imagen:etiqueta'
        format: 'table'
        exit-code: '1'
        ignore-unfixed: true
        severity: 'CRITICAL'

    1. OWASP Juice Shop¶

    # Iniciar con Docker
docker run --rm -p 3000:3000 bkimminich/juice-shop

# Luego navegar a http://localhost:3000

    2. DVWA (Damn Vulnerable Web Application)¶

    # Iniciar con Docker
docker run --rm -it -p 80:80 vulnerables/web-dvwa

# Luego navegar a http://localhost
# Credenciales predeterminadas:
# Usuario: admin
# Contraseña: password

    3. WebGoat¶

    # Iniciar con Docker
docker run -it -p 8080:8080 -p 9090:9090 -e TZ=Europe/Madrid \
  -e WEBGOAT_ADMIN_PASSWORD=CambiaEsto! \
  --name webgoat webgoat/goatandwolf:v8.2.2

# Luego navegar a http://localhost:8080/WebGoat

    Ejercicio práctico¶

    1. Configura GitHub Actions en tu proyecto para realizar análisis de seguridad automatizados
    2. Ejecuta OWASP ZAP contra tu aplicación local y soluciona los problemas encontrados
    3. Utiliza Snyk o Dependabot para analizar la seguridad de las dependencias de tu proyecto
    4. Instala SonarLint en tu IDE y corrige los problemas detectados

    Consejos finales¶

    • La seguridad es un proceso, no un producto: Actualiza y revisa tu aplicación continuamente
    • "Seguro por defecto": Integra la seguridad desde el inicio del desarrollo
    • Principio de mínimo privilegio: Usuarios y procesos deben tener solo los permisos necesarios
    • Múltiples capas de defensa: No confíes en una sola medida de seguridad
    • Establece una política de seguridad clara: Tu equipo debe saber qué está permitido y qué no
